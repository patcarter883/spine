# Coding Guidelines

## Logging Conventions

The project relies on Python's standard `logging` module and follows two complementary rules: a single, centralized function configures the root logger, and every module that emits logs obtains its own logger via `logging.getLogger(__name__)`. This keeps log levels, handlers, and output consistent while making the source of each message traceable. Contributors must not bypass these patterns with ad-hoc `print` calls, direct `logging.warning`-style module functions, or one-off root-logger mutations outside the established configuration helper. The evidence for this convention is found in `spine/log.py`, which owns the setup logic, and `spine/config.py`, which demonstrates module-level logger usage inside `SpineConfig`. All logging setup has observable side effects, so it must be invoked at application startup and treated as part of the runtime bootstrap rather than scattered through business logic. New log levels or output targets should be added by extending the existing configuration function and the module-logger pattern, not by introducing parallel logging machinery. The current codebase only configures `DEBUG` and `WARNING` levels and only shows `.warning()` calls on a module logger, so contributors should treat that as the supported surface until the guideline is explicitly expanded. Any change that introduces additional levels, handlers, or direct `logging.*` calls must also update `spine/log.py` and the module-level logger acquisition pattern to remain consistent with the established convention. Logging is not a free-form diagnostic tool in this codebase; it is a managed, typed part of the bootstrap contract, and every log emission must be routed through a logger named after the module that produces it. This rule is grounded solely in the supplied fragment and the symbols it references: `configure_logging` in `spine/log.py` and `SpineConfig` in `spine/config.py`. No other logging modules, helper classes, or third-party logging frameworks are evidenced and therefore must not be assumed or introduced without a separate guideline update. Contributors who add new packages should mirror the `spine/config.py` pattern: import `logging`, create a module-level logger with `logging.getLogger(__name__)`, and use that logger for all runtime diagnostics. Startup code should call `configure_logging` once before any module-level logger is expected to produce output, ensuring that the root logger level and handler are already in place. This avoids lost messages, duplicate handlers, or inconsistent level filtering across the application. The convention is intentionally small and enforceable: central configuration plus per-module named loggers, with no exceptions for convenience logging. Reviewers should reject changes that create loggers with hard-coded strings, call `logging.basicConfig` outside `configure_logging`, or emit logs through the global `logging` module functions. Keeping logging disciplined preserves the ability to reason about output, test log behavior, and control verbosity from a single configuration point. It also ensures that future observability integrations—such as structured formatters or external handlers—can be added in one place rather than retrofitted across dozens of modules. Because the supplied fragment is the only source of truth, every rule below is tied directly to the observed calls and side effects in `spine/log.py` and `spine/config.py`. Do not infer conventions from other files, languages, or frameworks; if a new logging need arises, update this guideline with fresh evidence before deviating from the patterns shown here. The goal is to make logging predictable, reviewable, and mechanically enforceable: one configuration function, one logger per module, and explicit levels that are declared in the central configuration code. ## Data Model Conventions

The codebase follows a **frozen dataclass** rule for internal data models. Any object whose primary purpose is to carry structured analyzer state must be immutable and declared with `@dataclass(frozen=True)`. This convention is visible in the analyzer pipeline, where the public `RepoAnalyzer` class manipulates value-object containers such as `SpineConfig` and `RepoManifest`, plus nested records like `ModuleBoundary`, `DependencyEdge`, and `PatternFinding`. Treating these as frozen dataclasses keeps analysis outputs hashable, comparable, and safe to pass between caching, logging, and serialization layers without accidental mutation. Contributors must extend this pattern when they add new structured data carriers in the same domain.

### Analyzer data models (`spine/work/onboarding/analyzer.py`)

Internal data carriers used by `RepoAnalyzer` must be frozen dataclasses. Evidence shows `SpineConfig`, `RepoManifest`, `ModuleBoundary`, `DependencyEdge`, and `PatternFinding` are used as structured value objects that compose the repository manifest. Add new model classes here (or in any equivalent analyzer module) only as `@dataclass(frozen=True)` with typed fields; do not attach mutable caches, file handles, or side-effecting methods to them.

**Notes:**
- The evidence only establishes this convention for the analyzer’s manifest/config objects. It does not dictate the implementation of persistence, deserialization, or test fixtures, so those layers should be evaluated separately before applying the same rule.
- `tests/unit/test_read_cache_dedup.py` is referenced in the findings but provides no `data_model` evidence, so no rule about cache deduplication data models can be inferred from this fragment.

## Error Handling Conventions

All fallible operations must be guarded with `try/except` blocks. When an external dependency, encoding call, or filesystem operation can fail, code must recover through a well-defined fallback and surface the failure via logging rather than allowing an unhandled exception to propagate. The established pattern across the codebase is defensive: anticipate failures, degrade gracefully, and record warnings so operators can diagnose issues without crashing the calling workflow.

This convention applies to public API functions that depend on third-party libraries, filesystem traversal, or any other operation with observable failure modes. Fallbacks must remain consistent with the function's documented purpose — for example, a token-counting helper falls back to a character-based estimate, and an artifact scanner continues to return whatever portion of the directory it can safely read.

### try/except guards around fallible operations

The project-wide `error_handling` convention. Contributors must wrap any operation that can raise in a `try/except` guard, supply a fallback behavior that preserves the function's contract, and log warnings when recovery occurs.

### count_tokens

Wraps `tiktoken.get_encoding` and `enc.encode` in a `try/except` guard and falls back to a character-based token estimate when tiktoken fails, keeping the public API resilient to encoding errors.

### scan_artifact_dir

Guards filesystem operations such as `iterdir`, `is_file`, and `read_text`; emits a `logger.warning` on failure and returns a dict of filenames to truncated content previews, allowing partial artifact discovery to continue.

### agent_adversarial_check

Applies `try/except` guards around fallible operations during adversarial review, following the same defensive pattern established in the other listed functions.

**Notes:**
- The supplied fragment does not detail a specific fallback for `agent_adversarial_check`; contributors should mirror the guard-plus-fallback pattern shown in `count_tokens` and `scan_artifact_dir`.

## Testing Conventions

All tests in this project are written with **pytest**. Contributors must place test code inside the `tests/` directory, use `test_*.py` files, and define test functions whose names start with `test_`. Class-based test suites must use class names beginning with `Test`. Integration tests should be grouped under subdirectories such as `tests/integration/` rather than mixed with the package source code. Shared fixtures and pytest configuration must live in `tests/conftest.py` so they are discovered automatically by pytest without being imported explicitly.

### pytest discovery and naming (`tests/`)

All pytest test functions must be named `test_*`, and class-based tests must use `Test*` class names. Test modules must be named `test_*.py` and placed under `tests/`.

### shared pytest configuration (`tests/conftest.py`)

Project-wide pytest helpers, fixtures, and configuration must be defined in `tests/conftest.py` (e.g., `test_config`) and are discovered automatically by pytest.

### integration test layout (`tests/integration/`)

Integration tests must be organized under `tests/integration/` rather than alongside source code, making the test suite structure match the subsystem separation shown by files such as `test_exploration_subgraph.py`.

### integration test example (`tests/integration/test_exploration_subgraph.py`)

Integration tests must expose behavior through focused `test_*` functions, such as `test_exploration_subgraph_builds` and `test_exploration_subgraph_supports_plan`, validating distinct capabilities of the component under test.

**Notes:**
- The supplied fragment only shows function-based tests; a class named `Test*` is allowed by convention but not currently evidenced in the sample.
- Do not add tests outside `tests/` or use non-standard naming (e.g., `*_test.py`), because pytest will not match the project's established discovery pattern.

## Config Conventions

Configuration in this project must be accessed through a single, centralised mechanism: `SpineConfig.load()`. Contributors should not read environment variables, parse config files, or define ad-hoc defaults inside individual agent modules. Instead, load settings once with `SpineConfig.load()` and propagate the resulting configuration object to the code that needs it. This keeps runtime behavior predictable, makes settings discoverable, and simplifies testing by giving one obvious seam for injecting test values.

### Centralised `SpineConfig.load()` access (`spine/agents/decomposer.py`)

Agent modules such as `spine/agents/decomposer.py` must follow the centralised configuration convention. For example, `run_decomposer` orchestrates LLM-driven decomposition and should obtain any model, retry, logging, or mode settings through `SpineConfig.load()` rather than collecting them from scattered sources. Any new configurable parameter added to the decomposer must be routed through the same central config object so the function remains consistent with `PLAN`, `FALLBACK`, and `PER_FILE` mode handling.
